# Ring LWE

Now that we've got the basics down, let's introduce a more useful scheme: Ring LWE.

The good news is that the formulas or the mechanics themselves basically don't change a bit.

But there are still many things to be learned here

# Disadvantages of LWE

We have touched upon this at the end of chapter 01, but we did not mention all of them. So here is why simple LWE just doesn't cut it.

## Size of public matrix

The public matrix is huge. With `n=256`, it contains 65536 integers at least 16bit wide. And even though we don't need to **transmit** the whole matrix, we still need to **store** it in memory.

Right now, it would take about 128 kB. It doesn't seem that big, but does the chip in your wireless headphones have 128 kB of free memory to spare just to store a key?

## Transmission rate

Simple LWE can only encapsulate 1 bit at a time. If we want to encapsulate the standard 32 byte key, we would need 256 different ciphertexts. Each ciphertext stores a vector and a number.

Because of that we once again have 256x a vector with 256 elements. And that is, once again, another 128 kB of memory, not to mention you actually need to **transmit** these 128 kB.

# Enter Ring LWE

Unlike the simple LWE, which operated on matrixes and vectors, Ring LWE operates on polynomials. Polynomial is basically an expression that looks like this:

$a + b x^1 + c x^2 + d x^3 + ...$

*How do you store this in memory?*

You just need the coefficients `(a, b, c, d, ...)`. It is literally the same data structure as a vector, but the operations it supports are different.

## The keygen, encaps and decaps

Now let's see, how the formulas changed.

We are still using

`q = 3329`
`n = 256`
`k = 3`

## Keygen

Alice samples the following:

- Public pseudorandom polynomial `a` of length `n` and numbers `0..q-1`
- Secret polynomial `s` of length `n` and numbers `-k..k`
- Error polynomial `e` of length `n` and numbers `-k..k`

And then you compute the Public polynomial `t`

$t = a s + e$

Looks familiar?

The public key is the tuple `(a, t)`

**Note**: Now the ugly large matrix is gone and has been replaced by a simple polynomial with only 256 coefficients

## Encapsulation

Bob samples the following

- Ephemeral secret polynomial `r` of order `n` and numbers `-k..k`
- Small error polynomials `e1` and `e2` of order `n` and numbers `-k..k`

He also encodes the message `m`

### Encoding

The encoding does not change at all. It still works on mapping `0 -> 0` and `1 -> q/2`.

What changes is the fact that message `m` is now a **polynomial**, so the whole 32 byte key fits in the 256 coefficients of the polynomial

### Encryption

And I guess nobody is surprised by the following formulas

We compute the Key polynomial `u` like this:

$u = a r + e_1 \pmod q$

And then we also compute the Ciphertext `v`

$v = r t + e_2 + m \pmod q$

## Decapsulation

Alice now computes the following:

$v - u s \pmod q$

Which yields a polynomial with coefficients that are **approximately** `0` or `q/2` and therefore the bit `0` or `1`.

## Polynomial multiplication - Ring

When we multiplied two vectors (dot product), we ended up with a simple integer

When we multiply two polynomials, we end up with another polynomial

$pol_1 = 1 + 2 x + 3 x^2$

$pol_2 = 2 + 3 x + 1 x^2$

Now the multiplication yields:

$pol_3 = 2 + 7 x + 13 x^2 + 11 x^3 + 3 x^4$

Wow, we ended up with 5 coefficients instead of 3! What now?

Well, there is one last thing I didn't mention. The polynomials are interpreted with $x^3 = -1$. What does that mean? Let's apply this to our example equation above.

The start remains the same

$2 + 7 x + 13 x^2$

But then

$11 x^3 = 11 * -1 = -11$

Our $x^3$ term has changed into $-1$ and this works similarly for the $x^4$ as well

$3 x^4 = 3 * -x = -3 x$

Do you see now? It **wraps around**, that's what we mean by the **Ring**.

In other words the equation above has the following solution in our $x^3 = -1$ ring:

$pol_3 = 2 + 7 x + 13 x^2 + 11 x^3 + 3 x^4 = -9 + 4 x + 13 x^2$

And now we are back to our original 3 coefficients

**Note**: The real scheme works with $x^{256} = -1$

## NTT transform multiplication

There is just one issue. The polynomial-polynomial multiplication is costly. We need to make it faster.

That's why real schemes use something called NTT multiplication

In hindsight it works like this:

1. We have two polynomials `a` and `b` and we want to compute `c = a * b`
2. We take the NTT transform of those polynomials `A = NTT_Transform(a)` and `B = NTT_Transfrom(b)`
3. We multiply those transforms element-wise (`A[0] * B[0]`, `A[1] * B[1]`, ...)
4. After that we apply the Inverse NTT transfrom `c = InvNTT(A * B)`, and we are done!

This process is much faster. But what are those mysterious NTT functions really?

If you are interested, check out chapter `02-1. NTT`.

## Summary

In summary, Ring LWE solves many problems of the simple LWE. But even then, as of writing this document, pure Ring LWE (one polynomial) is not used in any NIST-approved scheme. Why?

It all comes down to the Rings. Cryptoanalysts are afraid that the structure that Rings provide could be used against the scheme. (Although no proof exists yet, with cryptography one must be vigilant).

So what do we have?

| Simple LWE |
| ---------- |
| Very secure (basically no structure) |
| Computationally expensive and impractical |

| Ring LWE |
| -------- |
| Has ring structure enforced (might lead to cryptoanalytic attacks) |
| Fast and compact |

It would be nice if there was some option, which would be a compromise between the two, would it not? (slight foreshadowing)
